Deep Agents CLI — From the Outside In
Module 1

What This App Does

You talk to an AI in your terminal. Here's what actually happens when you hit Enter.

A coding assistant that lives in your terminal

Deep Agents CLI is a CLI tool that gives you an AI coding assistant powered by a large language model. Instead of opening a website, you type in your terminal and the AI reads files, writes code, runs shell commands, and searches the web — all as your user, on your machine.

💬 Chat with AI
📁 Read & write files
⚡ Run shell commands
🔍 Search the web
🧠 Persistent memory
💡
Why does this matter?
Unlike a chatbot in a browser, this AI has access to your actual project files. It can understand the code you're working on, make changes directly, and run tests — not just suggest what to write.

The journey of a single message

Every time you type a message and press Enter, a carefully choreographed sequence fires up. Here's the ballet behind the scenes.

1
You type a message
Something like "add a login button to the settings page." The CLI captures your keystrokes in a text input widget.
2
The message enters a queue
If the AI is already busy, your message waits. The queue makes sure the AI doesn't get overwhelmed by a flood of rapid requests.
3
The AI reads the message and thinks
The AI model (running on a server somewhere, not your machine) reads your message along with your entire conversation history and project context.
4
The AI decides to use tools
It might call a tool like "read this file" or "run this command." Each tool call pauses the AI's response until the tool returns a result.
5
Results stream back to you
The AI's response — and any tool results — appear on screen in real time, character by character, like someone is typing.

What the entry point looks like

When you run deepagents, this is the first code that runs. The CLI decides whether to show the interactive app or handle a one-off command like list or reset.

deepagents_cli/main.py — simplified
1def main() -> int: 2 args = parse_args() 3 if args.command == "run": 4 run_app(args) # ← starts the TUI 5 elif args.command == "list": 6 list_agents(args) # ← prints agent list 7 elif args.command == "reset": 8 reset_agent(args) # ← resets prompt 9 return 0
Line 1 — Defines the main function. Everything starts here.
Line 2 — Reads your command-line arguments (flags like --agent, -r).
Line 3–4 — If you typed just deepagents (no subcommand), fire up the interactive app.
Line 5–6 — If you typed deepagents list, print all known agents.
Line 7–8 — If you typed deepagents reset, reset the agent's prompt file.
Line 9 — Return 0 means "success." Any other number means an error happened.

Meet the Actors

Eight key characters work together to make this tool run. Knowing who does what helps you tell the AI where to put things.

The cast of characters

Think of the CLI as a movie production. Each "actor" has a specific role. When you talk to the AI, you're directing the whole cast — but the AI needs to know which role to assign each task.

🎬
main.py
The director. Decides whether to launch the interactive app or handle a subcommand like list or reset. Runs first, exits fast.
🖥️
app.py
The stage manager. Runs the Textual TUI — the interactive window, the chat history, the input box. Keeps everything rendered on screen.
🤖
agent.py
The producer. Builds the AI agent — wires up the language model, tools, memory, and middleware. The architect of what the AI can do.
🔌
textual_adapter.py
The translator. Converts between Textual UI events and LangGraph agent commands. Handles streaming, interrupts, and the approval menu.
⚙️
config.py
The stage crew. Handles settings, API keys, glyphs (symbols for the UI), and shell command allow-lists. Everything the app needs to know about its environment.
🛠️
tools.py
The toolbox. Provides the agent's web search (via Tavily) and URL fetching tools. Optional — only active if API keys are configured.
🧠
skills/
The rehearsal notes. Skills are like custom instructions that augment the AI — extra capabilities you can load from files on disk. The middleware loads and manages them.
📝
command_registry.py
The teleprompter. Defines all slash commands (/clear, /model, /threads, etc.) and their priority tiers — which ones can interrupt the AI mid-thought.

Module 3

How the Pieces Talk

Data flows through the system in predictable patterns. Understanding the flow helps you debug "why isn't X showing up?"

The message queue — traffic control for the AI

Imagine the AI as a busy executive. If five people send messages at once, they don't all interrupt — they wait in a queue. The AI processes them one at a time.

🚦
The metaphor: an air traffic controller
When planes line up to land, the controller doesn't let them all descend at once — they stack in a queue and get clearance one at a time. The message queue works the same way: it keeps requests orderly and prevents the AI from getting overwhelmed.
1
You type a message
The Textual input widget captures it and submits it to the app's message bus.
2
The queue decides priority
Some messages are always-bypass (/quit). Others are queued (/clear). The queue classifies each message and either lets it through immediately or makes it wait.
3
The agent processes one at a time
While the agent is thinking or running tools, new messages wait. Once the agent is idle, the next message is pulled from the queue.

The Human-in-the-Loop safety valve

Before the AI can run certain "dangerous" operations — writing files, running shell commands, searching the web — it pauses and asks your permission. This is called HITL.

🤖
Agent
🛑
HITL Interrupt
👤
You (approve/reject)
⚡

🔐
What gets gated?
Shell commands (execute), file writes/edits (write_file, edit_file), web search, URL fetching, and subagent delegation are all behind the HITL gate. If you run with -y / --auto-approve, all gates open automatically.

The middleware pipeline

Middleware is like a series of filters that every AI request and response passes through. Each one can inspect, modify, or act on the data.

How middleware is stacked (simplified from agent.py)
1agent_middleware = [] 2agent_middleware.append(ConfigurableModelMiddleware()) 3agent_middleware.append(AskUserMiddleware()) # ← AI asks you questions 4agent_middleware.append(MemoryMiddleware()) # ← persistent memory 5agent_middleware.append(SkillsMiddleware()) # ← loaded skills 6agent_middleware.append(LocalContextMiddleware) # ← git info, file tree 7agent_middleware.append(SummarizationMiddleware) # ← context window management
Line 1 — Start with an empty list of middleware.
Line 2 — ConfigurableModel: wraps the language model so it can be swapped or configured at runtime.
Line 3 — AskUser: gives the AI a tool to ask you clarifying questions mid-task.
Line 4 — Memory: loads persistent context from AGENTS.md files so the AI remembers past conversations.
Line 5 — Skills: injects custom instructions from skill files on disk.
Line 6 — LocalContext: provides git branch, directory tree, and project info to the AI.
Line 7 — Summarization: compresses old messages when the context window gets full.
🧩
Middleware ordering matters
Each middleware sees the output of the one before it. Memory is added before Skills so that skill loading can reference memories. AskUser is added early so its tool is available for the whole conversation.

The Outside World

The CLI doesn't run in isolation — it talks to language models, external APIs, MCP servers, and optional sandbox clouds.

Language models — the brain

The AI "brain" isn't running on your machine — it's a remote language model API. The CLI supports multiple providers.

🧠
Anthropic Claude
Set ANTHROPIC_API_KEY. Specify model with --model anthropic:claude-sonnet-4-6.
🤖
OpenAI GPT
Set OPENAI_API_KEY. Specify with --model gpt-4o.
🌴
Google Gemini
Set GOOGLE_API_KEY or use Vertex AI via GOOGLE_CLOUD_PROJECT.
🔑
How models are configured
The model_config.py file maps provider names to environment variables. When you pass --model anthropic:claude-sonnet-4-6, the CLI looks up ANTHROPIC_API_KEY and creates the appropriate LangChain model client. Adding a new provider means updating that mapping.

MCP servers — plugging in external tools

MCP lets the CLI connect to external tool servers. Think of it like plugging a USB device into your computer — the tool appears as if it were built-in.

1
The CLI finds MCP configs
It scans for mcp.json or mcp.yaml files in your project and home directory, plus any path passed via --mcp-config.
2
Servers are started
Each server is spawned as a subprocess. The CLI communicates with it over stdio.
3
Tools become available to the AI
The AI sees MCP tools alongside its built-in tools. You can approve or reject individual MCP tool calls in the HITL menu.

Sandboxes — running code safely in the cloud

By default, the AI runs shell commands on your machine. But you can point it at a remote sandbox — useful for dangerous operations or cross-platform testing.

Local vs. Remote — the key difference
In local mode, the CLI's LocalShellBackend runs commands on your machine. In sandbox mode, the CLI sends commands to a remote Linux VM and streams results back. The UI and UX are nearly identical — but your files are on the remote machine, not yours.

Module 5

The Clever Tricks

Several smart patterns make this CLI feel polished: lazy loading, context summarization, unicode security, and a smart shell allow-list.

Lazy loading — keeping startup fast

The CLI must launch quickly — even for a simple deepagents --version. Heavy imports (the AI SDK, LangChain, Textual) are deferred until they're actually needed.

config.py — bootstrap pattern
1def _ensure_bootstrap() -> None: 2 global _bootstrap_done 3 if _bootstrap_done: return 4 with _bootstrap_lock: 5 if _bootstrap_done: return # double-check 6 _load_dotenv() # read .env file 7 _override_langsmith_project() 8 _bootstrap_done = True
Line 3 — If bootstrap already ran, exit immediately. No work needed.
Line 4 — Thread lock: prevents two threads from running bootstrap simultaneously (important in a Textual app where background workers may start concurrently).
Line 5 — Double-checked locking: after acquiring the lock, check again. The first thread sets the flag; subsequent threads skip everything.
Line 6 — Read the .env file from the project root (or cwd) to load API keys and config.
Line 7 — Override the LANGSMITH_PROJECT env var so agent traces go to a separate LangSmith project from the user's own traces.
Line 8 — Mark bootstrap done. Future calls exit at line 3.
⚡
The startup hot path
deepagents --help and deepagents --version should be near-instant. They trigger zero SDK imports. The Textual app itself only imports heavy packages after the splash screen renders.

Context window management — the "offload" trick

AI models have a maximum context window — a limit on how many tokens (words, roughly) they can see at once. Long conversations eventually hit that limit. The CLI handles this with a summarization middleware.

1
Older messages are offloaded
When context gets tight, older messages are sent to a separate storage backend (/conversation_history/) and replaced with a compact summary.
2
The AI gets the summary
The model sees a short recap instead of the full history, freeing up context space for new work.
3
Full history is preserved
The actual conversation is stored in a virtual filesystem. It's retrievable — just not visible to the model unless explicitly fetched.

Unicode security — hiding malicious commands

Some characters look identical but are actually different — an attacker could hide a dangerous command inside what looks like a harmless string. The CLI scans for this.

🔍
Detection
The detect_dangerous_unicode() function scans shell commands for homoglyph attacks — characters that look like normal letters but are actually from a different Unicode script.
⚠️
Warning
If detected, the command is shown with the hidden characters revealed using visual markers (red highlighting on the dangerous codepoints).
🚫
Blocking
Commands with certain dangerous patterns — $(...), backticks, redirects — are blocked entirely, even if the base command is on the allow-list.

Check your understanding

Why does the CLI use a double-checked locking pattern in _ensure_bootstrap()?
Exactly! The first check (outside the lock) lets most calls exit instantly. The second check (inside the lock) prevents two threads from both seeing _bootstrap_done = False and running bootstrap concurrently. This is a standard concurrency pattern.
The lock actually slows things down slightly. The benefit is safety, not speed.
Reading .env twice would be a bug, not a feature. The double-check prevents concurrent execution.
Module 6

When Things Break

Knowing how to debug means knowing where to look. Here's the map.

The debugging map

When something goes wrong, the first question is: where did it happen? Here's where to look for different problem types.

🤖
AI gives a bad response
Check the system prompt in agent.py's get_system_prompt(). Also check the model's AGENTS.md file for conflicting instructions. Try /clear to start a fresh conversation.
⛔
Tool call denied
Check the HITL configuration in _add_interrupt_on() in agent.py. Is the tool listed there? Is auto_approve mode on?
🐛
Tool doesn't work
Check the tool definition in the SDK (not the CLI). The CLI wraps tools from deepagents. Run deepagents -n "list your tools" to see what the agent actually has.
🔑
API key errors
Check config.py's Settings.from_environment(). API keys should come from environment variables. Make sure the right variable is set for your provider.
🔀
Slash command not found
Check command_registry.py's COMMANDS tuple. Is the command listed there? Also check the bypass_tier — if it's wrong, the command might not be accessible in the current context.
📝
Memory not working
Check the MemoryMiddleware setup in create_cli_agent(). Make sure AGENTS.md files exist at the expected paths. Run deepagents list to see known agents.

LangSmith — the flight recorder

Every agent run is recorded in LangSmith if LANGSMITH_TRACING is enabled. You can replay conversations, see tool calls, measure token usage, and spot where the AI went off-track.

🔗
The /trace slash command
While in the CLI, type /trace to open the current thread's LangSmith trace in your browser. This is the fastest way to inspect what the AI actually did — every tool call, every decision, every token.
📌
The project separation trick
The CLI overrides LANGSMITH_PROJECT so agent traces go to a separate project from your own code's traces. This keeps the "noise" of your development work separate from the AI's decision trail.

Check your understanding

A user reports: "The AI keeps ignoring my instructions in AGENTS.md." Where do you start debugging?
Right! The full debugging chain is: (1) file exists at the right path, (2) MemoryMiddleware is pointing to it, (3) the content actually says what you think it says. Any one of these failing breaks the chain.
That's part of it, but there are two other possible failure points — the middleware and the content itself.

Why it's built this way

These aren't accidental — each decision solves a specific problem.

Textual for the TUI
Textual was chosen over alternatives (Blessed, npyscreen) because it has first-class async support, reactive state management, and a modern widget model. The learning curve pays off in maintainability.
LangGraph under the hood
The SDK uses LangGraph (from LangChain) to manage the agent's state machine. This handles tool calling, retries, checkpoints, and streaming — things you'd otherwise have to build from scratch.
CompositeBackend routing
Large tool outputs and conversation history are routed to temp directories so they don't pollute the user's working directory. The backend is composable — different paths go to different storage backends.
Skills as files on disk
Skills are Markdown files loaded at runtime, not hardcoded instructions. This means non-engineers can add skills by dropping a file in a directory — no code changes needed.
